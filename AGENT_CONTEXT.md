# AGENT_CONTEXT.md

## Objective
이 문서는 AI 코딩 에이전트가 현재 프로젝트에서 작업할 때 따라야 할 목표, 범위, 제약조건, 검증 방식, 보고 형식을 정의한다.
에이전트는 이 문서를 먼저 읽고, 여기 적힌 규칙을 우선 적용한다.

## Task Goal
이번 작업의 목표:
- [여기에 작업 목표를 적는다]

예시:
- 로그인 실패 횟수 5회 제한 기능 추가
- 특정 API 응답 속도 개선
- 결제 실패 로그를 더 명확히 남기도록 수정

## In Scope
에이전트가 수정해도 되는 범위:
- [수정 가능한 디렉토리/파일]
- [수정 가능한 모듈]
- [허용되는 작업 종류]

예시:
- `src/auth/**`
- `src/api/login.ts`
- 관련 테스트 파일 수정 가능

## Out of Scope
에이전트가 수정하면 안 되는 범위:
- [수정 금지 디렉토리/파일]
- [건드리면 안 되는 기능]
- [이번 작업과 무관한 리팩터링]

예시:
- DB 스키마 변경 금지
- 인프라 설정 변경 금지
- 인증 외 다른 API 동작 변경 금지
- 관련 없는 코드 정리 금지

## Constraints
반드시 지켜야 하는 제약조건:
- 기존 코드 스타일과 패턴을 우선 따를 것
- 최소 수정 원칙을 따를 것
- 사용자 요청 없는 신규 의존성 추가 금지
- 사용자 요청 없는 공개 API 변경 금지
- 사용자 요청 없는 대규모 리팩터링 금지
- 기존 테스트가 있으면 우선 활용할 것
- 확인하지 않은 구조를 추정해서 구현하지 말 것

프로젝트별 제약:
- [여기에 추가]
- [예: TypeScript strict 모드 유지]
- [예: React Query 사용 패턴 유지]
- [예: 클래스 컴포넌트 도입 금지]

## Anti-Hallucination Rules
에이전트는 아래 규칙을 반드시 지킨다.

- 확인하지 않은 파일, 함수, 클래스, API, 환경변수, 설정값을 존재하는 것처럼 말하지 않는다.
- 열어보지 않은 파일 내용을 아는 척하지 않는다.
- 실행하지 않은 테스트/빌드/린트 결과를 성공했다고 쓰지 않는다.
- 확실하지 않은 내용은 `Assumption` 또는 `Unverified`로 표시한다.
- 정보가 부족하면 억지로 추정해 채우지 말고 부족한 점을 명시한다.
- 최신 정보가 필요한 내용은 확인 없이 단정하지 않는다.
- 작업 완료 보고는 반드시 증거 기반으로만 작성한다.

## Working Rules
에이전트는 아래 순서로 작업한다.

1. 요구사항을 짧게 요약한다.
2. 관련 코드, 설정, 테스트, 문서를 먼저 확인한다.
3. 범위 내에서만 구현한다.
4. 가능한 검증을 수행한다.
5. 결과를 지정된 형식으로 보고한다.

## Acceptance Criteria
작업 완료로 간주되는 조건:
- [기능적 완료 조건]
- [호환성 조건]
- [테스트 조건]
- [성능/보안 조건]

예시:
- 로그인 5회 실패 시 계정이 잠긴다.
- 기존 성공/실패 응답 JSON 형식은 유지된다.
- 관련 테스트가 모두 통과한다.
- 타입체크 오류가 없다.

## Validation
작업 후 가능한 범위에서 아래 검증을 수행한다.

필수 검증:
- [예: `npm test`]
- [예: `npm run lint`]
- [예: `npm run build`]
- [예: `npm run typecheck`]

선택 검증:
- [예: 특정 테스트 파일만 실행]
- [예: 로컬 재현 절차 수행]

검증 규칙:
- 실행한 것만 보고한다.
- 실행 못 한 것은 이유와 함께 명시한다.
- 실패한 검증은 숨기지 않는다.

## Output Format
에이전트의 완료 보고는 아래 형식을 따른다.

### Summary
- 무엇을 변경했는지 짧게 설명

### Changed Files
- 변경한 파일 목록

### Evidence
- 어떤 코드/설정/문서를 근거로 판단했는지 설명

### Validation
- 실행한 명령어
- 성공/실패 결과
- 실패 시 이유

### Assumptions
- 구현 중 둔 가정

### Unverified
- 확인하지 못한 사항
- 추가 점검이 필요한 부분

### Risks
- 잠재적 회귀 위험
- 후속 확인 포인트

## Completion Gate
아래를 모두 충족하기 전에는 완료라고 보고하지 않는다.

- 변경 파일 목록을 제시했다.
- 실제 수행한 검증 명령을 제시했다.
- 검증 결과를 성공/실패로 구분했다.
- 가정한 내용을 따로 적었다.
- 확인하지 못한 내용을 숨기지 않았다.

## Preferred Behavior
에이전트는 기본적으로 다음 행동을 선호한다.

- 큰 변경보다 작은 변경
- 새 패턴 도입보다 기존 패턴 재사용
- 추정보다 코드 확인
- 광범위한 수정보다 국소 수정
- 설명 없는 완료 보고보다 근거 포함 보고

## If Blocked
막히면 아래 형식으로 보고한다.

- Blocker: 무엇이 부족한지
- Why it matters: 왜 필요한지
- Safe fallback: 지금 가능한 가장 안전한 대안
- Unverified impact: 확인되지 않은 영향 범위

- [참고 구현 위치]
- [주의해야 할 예외]

## Example
예시 작업:
- 로그인 실패 5회 제한

예시 완료 보고:
- Summary: 로그인 실패 횟수 제한 로직을 추가했고 기존 응답 형식은 유지했다.
- Changed Files: `src/auth/service.ts`, `src/auth/service.test.ts`
- Evidence: 기존 인증 분기 로직과 테스트 패턴을 기준으로 구현했다.
- Validation: `npm test -- auth` 실행, 성공
- Assumptions: 실패 횟수 저장 위치는 기존 사용자 상태 모델을 재사용 가능하다고 가정
- Unverified: 멀티 인스턴스 환경에서 동시성 영향은 확인하지 못함
- Risks: 캐시 계층이 있다면 반영 타이밍 차이 가능성 있음
